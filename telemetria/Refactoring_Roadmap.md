## Refactoring Roadmap - Telemetria de Veículos

Como seu Staff Engineer particular, fiz um raio-x no projeto e, diva, temos uns pontos aqui que precisam de um carinho especial pra essa telemetria não virar um tele-pesadelo. O objetivo é transformar esse "vibe coded" em algo robusto, escalável e seguro. Sem enrolação, vamos ao que interessa:

| File/Location | Severity | Smell | Why Bad (explicação arch) | Priority Fix (instrução exata) |
|---|---|---|---|---|
| `setup/settings.py` | Critical/Safety | `DEBUG = True` em produção | Expõe informações sensíveis, rastreamentos de pilha e configurações internas, facilitando ataques e depuração não autorizada em ambiente de produção. | Mudar para `DEBUG = False` em produção e usar variáveis de ambiente para controlar. |
| `setup/settings.py` | Critical/Safety | `ALLOWED_HOSTS = []` | Em produção, permite que qualquer host acesse a aplicação, tornando-a vulnerável a ataques de cabeçalho HTTP Host. | Configurar `ALLOWED_HOSTS` com os domínios permitidos em produção. |
| `setup/settings.py` | Critical/Safety | `CORS_ALLOW_ALL_ORIGINS = True` | Permite requisições de qualquer origem, o que pode levar a ataques Cross-Site Request Forgery (CSRF) e vazamento de dados sensíveis se não houver outras camadas de segurança. | Restringir `CORS_ALLOWED_ORIGINS` para domínios específicos em produção. |
| `api_telemetria/worker.py` | Architectural/Performance | N+1 queries em `salvar_medicao` | Cada chamada a `salvar_medicao` faz duas consultas `objects.get` e uma `objects.create` para cada item, resultando em múltiplas requisições ao banco de dados para cada mensagem MQTT, o que é ineficiente para grandes volumes. | Implementar `bulk_create` para inserção em lote e pré-carregar `Veiculo` e `Medicao` para evitar N+1 queries. |
| `api_telemetria/worker.py` | Code Hygiene/Architectural | Duplicação de lógica de validação/persistência | A lógica de parsing e validação de dados é duplicada no `worker.py` e no `services.py`, sem reuso de serializers, aumentando a chance de inconsistências e bugs. | Refatorar para usar um serializer dedicado para a ingestão MQTT, centralizando a validação e persistência. |
| `api_telemetria/worker.py` | Code Hygiene/Architectural | Tratamento de erros com `print` | Erros são apenas impressos no console, sem um sistema de log adequado, dificultando o monitoramento e a depuração em produção. | Implementar logging estruturado (ex: `logging` do Python, Sentry) para registrar erros e eventos importantes. |
| `api_telemetria/worker.py` | Data/Security | Inconsistência no formato de data | O formato de data esperado (`%d/%m/%Y %H:%M:%S`) difere do formato de saída (`%Y-%m-%d`), e a conversão para `date` ignora informações de tempo. | Padronizar o formato de data/hora e garantir que a informação de tempo seja preservada se for relevante. |
| `api_telemetria/worker.py` | Data/Security | `MedicaoVeiculo.objects.create` sem transação | Cada inserção é atômica, mas o processamento de múltiplos itens não está em uma transação única, podendo levar a dados parciais em caso de falha. | Envolver o loop de inserção em uma transação atômica para garantir a consistência dos dados. |
| `api_telemetria/services.py` | Architectural/Data/Security | `executar_procedure_pos_importacao` (caixa preta) | A chamada a uma stored procedure dificulta a manutenção, teste, auditoria e portabilidade do código, além de acoplar a aplicação ao banco de dados. | Migrar a lógica da stored procedure para Python, dentro de um service layer. |
| `api_telemetria/services.py` | Data/Security | `MedicaoVeiculoTemp` com `on_delete=models.DO_NOTHING` | Se um `Veiculo` ou `Medicao` for deletado, os registros em `MedicaoVeiculoTemp` podem ficar órfãos, comprometendo a integridade referencial. | Alterar `on_delete` para `models.CASCADE` ou `models.SET_NULL` (com `null=True`) se a lógica de negócio permitir, ou implementar um tratamento explícito. |
| `api_telemetria/services.py` | Performance | Contagem redundante de `total_linhas_importadas` | A contagem é feita após `bulk_create`, sendo que `len(linhas_para_inserir)` já fornece o número de linhas válidas, gerando uma consulta desnecessária ao banco. | Remover a contagem redundante e usar `len(linhas_para_inserir)` para `total_linhas_importadas`. |
| `api_telemetria/views.py` | Critical/Safety | `permission_classes = [AllowAny]` | Permite acesso irrestrito a todas as APIs, o que é um risco de segurança grave em produção. | Implementar autenticação e autorização adequadas (ex: JWT, OAuth2) e restringir permissões. |
| `api_telemetria/views.py` | Architectural/Code Hygiene | Lógica de negócio nas views | As views `ImportarMedicaoCSVView` e `MedicaoVeiculoTempViewSet` contêm lógica de negócio que deveria estar em um service layer, tornando as views mais inchadas e menos reutilizáveis. | Mover a lógica de processamento de CSV e manipulação de `MedicaoVeiculoTemp` para um service layer dedicado. |
| `api_telemetria/models.py` | Architectural | `MedicaoVeiculoTemp` como modelo | O uso de um modelo de banco de dados para dados temporários de importação pode ser excessivo e gerar overhead. | Avaliar se um modelo temporário é realmente necessário ou se a importação pode ser feita diretamente para `MedicaoVeiculo` com validação robusta. |
| `api_telemetria/serializers.py` | Code Hygiene | Validações duplicadas | Algumas validações (ex: `validate_nome` em `MarcaSerializer` e `ModeloSerializer`) são repetitivas e poderiam ser abstraídas. | Criar uma classe base de serializer ou um mixin para validações comuns. |
